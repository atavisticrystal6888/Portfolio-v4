## Context & Problem

Plant enthusiasts and collectors struggle to diagnose plant-health issues early. Existing apps (Planta, PictureThis, Greg) lean on static care calendars and species lookups. None correlate visual symptoms with environmental history, and none cite their sources.

The gap: a forensic, auditable diagnosis — look at _your_ plant, factor in _your_ local conditions, and recommend an action grounded in botanical research, not generic templates.

> Source: [aarchid-rework](https://github.com/dfordp/aarchid-rework) · [aarchid-api](https://github.com/dfordp/aarchid-api) · live at [aarchid.space](https://www.aarchid.space/).

## Research & Discovery

- Interviewed 10+ plant enthusiasts about failure modes and trust gaps in existing apps
- Benchmarked top 4 apps on diagnostic specificity, citation quality, and edge-case handling
- Unmet need crystallised: image-based diagnosis + hyper-local environmental context + cited recommendations

## Solution & Approach

### 1. Architecture — the "Edge Stack"

Next.js PWA → Cloudflare Workers (Hono) orchestrator → Gemini 1.5 Pro (vision) + Perplexity Sonar (research) + OpenWeather (environment) → Supabase (Postgres) + Cloudflare R2 (image storage, zero egress). Result: sub-$5/mo fixed infra at low usage, ~$0.25 per active user at scale.

### 2. Forensic Health Audit

Upload photo → Gemini 1.5 Pro detects pests, deficiencies, stress → Workers call Perplexity Sonar to pull peer-reviewed citations → OpenWeather adds 7-day local context → response returns a health score (1–100), severity tier (Healthy / Warning / Critical), and a cited action plan.

### 3. Growth Velocity Tracking

Pixel-based measurement across photo logs tracks leaf expansion, stem elongation, and internodal distance over weeks. Species-specific benchmarks tell users whether a plant is thriving, stagnating, or declining.

### 4. Pro-sumer Asset Management

Batch care actions for 50+ plants at once grouped by "micro-climate zones". Exportable timestamped **Health Certificates** enable peer-to-peer plant sales with verifiable provenance — the business wedge.

## Implementation

The technical architecture prioritized modularity and fast iteration:

```python
# Simplified flow: Image → Analysis → Recommendation
from langchain.chains import SequentialChain
from langchain_google_genai import ChatGoogleGenerativeAI

vision_model = ChatGoogleGenerativeAI(model="gemini-pro-vision")
text_model = ChatGoogleGenerativeAI(model="gemini-pro")

# Step 1: Analyze plant image
diagnosis = vision_model.invoke([image, diagnosis_prompt])

# Step 2: Generate care plan from diagnosis
care_plan = text_model.invoke(care_prompt.format(
    diagnosis=diagnosis.content
))
```

> **Key Decision:** Using LangChain as an orchestration layer (rather than direct API calls) allowed rapid experimentation with prompt chaining and easy swapping of model versions.

## Learnings

### What Worked

Using LangChain's chain abstraction made it easy to iterate on prompts without touching infrastructure. The modular architecture let me develop and test each component independently.

### What I'd Change

Would add a feedback loop: let users confirm or correct diagnoses, then use that data to improve prompt engineering over time. The system is one-directional right now.
